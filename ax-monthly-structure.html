1) 임상시험/PMS SOP 문서 챗봇 구조 (NotebookLM 모니터링 범위 명시)

※ 현재 구조 기준으로는 NotebookLM 내부 질의·응답 전문의 중앙 모니터링은 제한적이며, 관리자 가시성은 계정/접근/운영 메타정보 중심입니다. 

flowchart LR
  subgraph Z1[사내 사용자 구간]
    U[임상팀 사용자]
  end

  subgraph Z2[서비스/운영 구간]
    A[NotebookLM 사용 환경\n사용자 직접 질의]
    M[관리자 가시영역\n계정/접근/운영 메타]
  end

  subgraph Z3[지식 구간]
    D[SOP 문서 소스\n약 120개 문서]
    O[운영담당자\n문서 변경 등록]
  end

  U -->|질의/검색| A
  A -->|응답 + 출처| U
  O -->|신규/개정 SOP 등록| D
  D -->|문서 업데이트 반영| A
  A -->|접근/운영 메타| M

  C1[[Control: RBAC/SSO 권한관리]]
  C2[[Control: 데이터 분류/업로드 기준]]
  C3[[Control: 운영 정책/감사 절차]]
  R1[[Risk: 질의·응답 전문 감사 한계]]
  R2[[Risk: 출처 표기 정확성]]

  A -.-> C1
  O -.-> C2
  M -.-> C3
  A -.-> R1
  A -.-> R2

3) 보안팀 검토 포인트 요약 (NotebookLM 기준)

구분 현재 확인 가능 현재 한계/불가 권장 통제/대안
사용자 활동 가시성 계정/접속/운영 메타 정보 중심 가시성 질의·응답 전문의 중앙 실시간 모니터링은 제한적 사내 프록시형 챗 UI 도입 시 감사로그(질의/응답/사용자/시간) 설계
데이터 통제 업로드 문서(SOP)와 변경 이력의 운영 관리 가능 문서 단위 세밀한 반출 감시/자동 차단은 별도 설계 필요 문서 분류등급, 업로드 승인 절차, 민감정보 비식별 기준 수립
접근권한 조직 계정 기반 접근통제(RBAC/SSO 정책 연동 가능 범위) 기능별 초세분 권한/결재 워크플로우는 운영정책 보완 필요 권한 매트릭스(사용자/운영자/관리자) + 정기 권한점검 시행
감사/사고대응 운영 로그 및 문서 변경 이력 기반 사후 점검 가능 질의 맥락 단위 정밀 포렌식은 데이터 수집 체계가 있어야 가능 로그 보관기간·점검주기·사고 보고체계(탐지→통보→조치) 문서화

권고: NotebookLM 단독 운영 시에는 “업무 보조 도구”로 범위를 명확히 하고, 중앙 감사 요구가 있는 경우 사내 중계 계층(프록시/게이트웨이) 또는 별도 챗 서비스로 확장 검토.
