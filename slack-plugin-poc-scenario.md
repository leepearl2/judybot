## 목적

Slack 하나에서 메일·결재·파일·그룹웨어 주요 기능을 처리할 수 있는지 검증한다. 플러그인 연동으로 기존 업무 방식과 동일한 흐름을 Slack 안에서 구현하는 것이 목표다.

| `#유형3_정보보호담당자_개인정보보호담당자` | 요청 집계 + 리포트 게시 |

## 섹션 1. 메일

### S1-1. Gmail → Slack 수신 알림

- Gmail 수신 이벤트를 Slack 채널에 자동 전달
- 전달 항목: 제목 / 발신자 / 수신 시각 / 요청 유형
- 운영자는 Slack 안에서 우선순위 판단 후 처리 플로우로 연계

### S1-2. Slack에서 Gmail 발송

- Slack 내 워크플로우 빌더로 메일 작성 및 발송 구현 가능
- 외부 탭 이동 없이 Slack 안에서 완결
- 발송 시 해당 채널에 발송 이력 자동 기록 (선택)

> :warning: *알려진 이슈 — 메일 제목 한글 인코딩*  
> 워크플로우 빌더를 통해 발송 시 메일 제목에 한글이 포함된 경우 인코딩 오류가 발생할 수 있음. PoC 기간 중 실사용 전 인코딩 정상 여부 반드시 검증 필요.

## 섹션 2. 결재

> 그룹웨어 = 전자결재 시스템으로 표기

### S2-1. 전자결재 요청 자동 접수

전자결재 요청 발생 시 워크플로우로 Slack 자동 접수

- 접수 시 고유 ID 부여 (`REQ-타임스탬프`), 상태 기본값 `open`
- 접수 즉시 담당 채널에 안내 메시지 발송

### S2-2. 처리 현황 추적

- 상태값 `open → done` 기반으로 진행 상황 추적
- 완료 시 완료 시각 기록, 이후 SLA 집계에서 자동 제외

### S2-3. SLA 초과 알림 (기준: 4시간)

- 접수 후 4시간 내 미처리 시 SLA 초과 판정
- 알림 시점에 과거 미해결 건도 누적 안내
- 채널별로 분리 알림 (해당 채널 건만 표시)

### S2-4. SSO / 링크 연동으로 그룹웨어 유사 경험 구현

EHR은 직접 연동 불가 → SSO + 링크 방식으로 현 그룹웨어와 유사한 접근성 확보

- Slack 북마크 또는 단축 명령어(`/결재`)로 그룹웨어 전자결재 페이지 즉시 접근
- 결재 대기 건 발생 시 Slack 알림으로 진입 유도

### S2-5. 일일 15시 리포트

- 매일 15:00(KST) 채널별 자동 리포트
- 포함 항목: 총 유입 / 처리 완료 / SLA 초과 / 미처리 리스트
- 각 채널에 해당 채널 건만 분리 게시

```text
[일일 운영 리포트 | 15:00]
채널: #유형1_정보보호팀 (또는 #유형2_ / #유형3_ 해당 채널명)

- 총 유입: {N}
- 처리 완료: {N}
- SLA 초과: {N}

[미처리 리스트]
- REQ-0001 | 요청제목 | 담당자 | 접수시각
```

## 섹션 3. 파일

### S3-1. Google Drive 파일 Slack에서 보기

- Drive 파일을 Slack 안에서 미리보기 및 열람
- 외부 탭 이동 없이 내용 확인 가능

### S3-2. Drive 변경 이벤트 Slack 알림

- 파일 생성 / 수정 / 공유 이벤트 발생 시 Slack 알림
- 알림 포함 항목: 문서 링크 / 변경 유형 / 변경자
- 변경 이벤트를 업무 요청과 연결해 추적 구성 가능

## 섹션 4. 그룹웨어 기타 기능 (추가 검토 항목)

현 그룹웨어에서 사용 중인 기능 중 Slack 연동 가능 여부 검토 대상:

| 기능 | 연동 방식 | 비고 |
|---|---|---|
| 공지사항 | Slack 채널 공지 대체 | 직접 구현 가능 |
| 조직도 / 사원 검색 | Slack 프로필 + 디렉토리 | 기본 제공 |
| 일정 관리 | Google Calendar 연동 | 플러그인 필요 |
| 문서 협업 | Google Drive 연동 | S3 시나리오 포함 |
| 업무 요청 / 티켓 | 워크플로우 자동 접수 | S2 시나리오 포함 |
| 전자결재 | SSO + 링크 연동 | S2-4 시나리오, EHR 직접 연동 불가, SSO 링크로 대체 |

## 운영 규칙

- 집계 대상은 워크플로우 접수건으로 한정한다.
- SLA 기준은 단일 기준(4시간)으로 운영한다.
- 리포트는 접수 채널과 동일한 채널에 분리 게시한다 (`#유형1`, `#유형2`, `#유형3` 각각).
- 상태값은 `open / done`만 사용한다.

## 데이터 모델 (Google Sheet 권장)

| 컬럼 | 설명 |
|---|---|
| `request_id` | 고유 ID |

| `completed_at` | 완료 시각 |

## 단계별 적용

| 단계 | 내용 |
|---|---|
| Phase 1 | SLA 기준 + 리포트 포맷 확정 |
| Phase 2 | 채널별 워크플로우 / 저장소 연결 |
